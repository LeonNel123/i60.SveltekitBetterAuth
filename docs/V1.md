## Existing Stack & Constraints

This project is **not** greenfield. It is built on an existing SvelteKit + Better Auth template. Respect and build on top of what already exists:

- **Framework:** SvelteKit 2, Svelte 5 (runes only ‚Äî `$state`, `$derived`, `$props`, `$effect`)
- **Auth:** Better Auth with plugins (organization, admin, twoFactor, emailOTP)
- **Database:** PostgreSQL 18.3 via Drizzle ORM (migrations with `drizzle-kit`)
- **UI:** shadcn-svelte components, Tailwind CSS v4 (CSS-first config in `src/app.css`, OKLch colours)
- **Email:** Pluggable provider (SendGrid configured, also supports Resend, SMTP, console)
- **Existing routes:** Auth flows (`/login`, `/register`, `/verify-email`, `/two-factor`, `/forgot-password`, `/reset-password`), settings (`/profile`, `/security`, `/organization`, `/members`), dashboard (empty shell)
- **Existing schema:** `user`, `session`, `account`, `verification`, `organization`, `member`, `invitation`, `two_factor` tables
- **Config:** `APP_NAME = 'BrokerOS'` in `src/lib/config.ts`

All new work goes into `src/routes/(app)/` for protected pages, `src/lib/server/db/schema.ts` for new tables, and `src/lib/components/` for new components. Read `AGENTS.md` for full API references and patterns.

## Core Concept

### CRM Command Centre Vision

A CRM system tailored to the needs of a South African short-term insurance brokerage.

---

### 1. Core Philosophy

#### 1.1 The CRM as a Command Centre

The system is not just a database of clients ‚Äî it is a **live operational command centre**. It should answer, at any moment:

- What is happening right now?
- What needs to happen next?
- What has already happened?
- Is anything being missed?

Every user ‚Äî from admin staff to business owner ‚Äî should be able to rely on the system as the **single source of truth**.

#### 1.2 Task-Driven Operations

At the heart of the system is a **task-driven model**:

- All meaningful work is represented as tasks
- Tasks can originate from:
  - User actions
  - Emails / inboxes
  - System triggers
- Tasks are assigned, tracked, escalated if necessary, and completed with traceability

The system behaves similarly to a **help desk / ticketing system**, but deeply integrated with client context.

---

### 2. Key Personas

#### 2.1 Business Owner / Broker

Needs:

- High-level overview of operations
- Visibility into outstanding work, missed commitments, and staff workload
- Early detection of bottlenecks, neglected clients, and SLA risks

#### 2.2 Administrative Staff

Needs:

- Clear view of what to do now, what is urgent, and what is overdue
- Easy access to client context, documents, and history
- Ability to create and update tasks, attach documents, and log notes

---

### 3. Core System Components

#### 3.1 Command Centre View (Dashboard)

The central interface of the system.

**Key features:**
- Personal task queue
- Team / shared task visibility
- Overdue and escalated tasks
- Upcoming scheduled work
- Workload distribution

**Purpose:** Provide a real-time operational snapshot of the business.

#### 3.2 Client (Insured) Entity

The foundational data structure.

**Types:**
- **Individual** ‚Äî South African ID number, personal details
- **Company** ‚Äî Registration number, business details

**Key capabilities:**
- Full history of interactions
- Associated policies
- Claims tracking
- Documents and attachments
- Notes and communication history

#### 3.3 Policies & Claims

Each client may have multiple policies and multiple claims.

**Important concepts:**
- Highlight or tag the "most active" or relevant policy
- Track claims lifecycle and associated work

#### 3.4 Documents & Attachments

A critical component for compliance and operations.

**Examples:** Policy schedules, renewal notices, inspection reports, bank letters, client-provided documents.

**Requirements:**
- Strong auditability (what was sent / received)
- Easy retrieval
- Association with client, policy, claim, or task

---

### 4. Document Tagging System

#### 4.1 Purpose

Enable flexible classification, fast filtering and search, and scalability without rigid schemas.

#### 4.2 Tagging Model ‚Äî Two-Tier System

1. **System tags (standardised):** Predefined, aligned with industry requirements. Examples: `policy-schedule`, `renewal`, `claim-document`, `kyc`, `inspection`.
2. **User-defined tags:** Custom and flexible ‚Äî extend the system without breaking structure.

#### 4.3 Tag Dimensions

Tags may represent:
- Document type
- Time period (e.g., `2025-01`)
- Version (`v1`, `v2`)
- Context (`claim-123`, `policy-abc`)

#### 4.4 Benefits

- Global document search
- Client-specific filtering
- Reduced reliance on rigid folder structures
- Adaptability to different broker workflows

---

### 5. Data Ingestion Strategy

#### 5.1 Initial Data Source

Assumption: An external system provides structured data extracted from policy schedules. This becomes the **seed data** for new clients.

#### 5.2 Folder-Based Import

To simplify onboarding:

- User selects a local or synced folder (e.g., OneDrive)
- System parses folder hierarchy, imports files, infers metadata (possibly via naming conventions), and attaches documents to the correct client

**Goal:** Reduce manual capture effort during client take-on.

> **V1 scope note:** Folder-based import is aspirational. For V1, implement manual client creation and individual file upload. The UI should accommodate future bulk import.

---

### 6. Tasks & Workflow

#### 6.1 Task Creation Sources

- Manual (user-created)
- Linked from client, policy, or claim context
- (Future: email / inbox ingestion, system triggers)

#### 6.2 Task Features

- Assignment (user or team)
- Reassignment
- Priority / urgency levels
- Due dates
- Escalation rules (overdue highlighting at minimum for V1)
- Followers (watchers)
- Status tracking

#### 6.3 Task Lifecycle

1. Created
2. Assigned
3. In progress
4. Completed
5. Audited / logged

#### 6.4 Relationship to Clients

Most tasks should be linked to a client. Tasks may also exist independently (e.g., internal admin tasks).

---

### 7. Notes & Activity History

Each client should have a chronological activity timeline including notes, tasks, emails, and document activity.

**Goal:** Any staff member can immediately understand what has happened, what is currently happening, and what needs to happen next.

---

### 8. Communication Integration

#### 8.1 Inbox Integration (Future ‚Äî Not V1)

Users should eventually be able to access personal and shared inboxes (claims, underwriting, general enquiries).

#### 8.2 Email-to-Action Flow (Future ‚Äî Not V1)

Incoming communication should allow: attach documents to client, create tasks, add notes, link to existing client automatically.

> **V1 scope note:** Email integration is out of scope for V1. The data model should be designed to accommodate it, but no inbox UI or email parsing is required.

---

### 9. Compliance & Regulatory Considerations

The system must support:
- South African regulatory requirements (FAIS, POPIA)
- Audit trails
- Document retention
- Traceability of communications, documents, and decisions

---

### 10. Design Principles

1. **Single source of truth** ‚Äî All operational knowledge lives in the system.
2. **Context over memory** ‚Äî No staff member should rely on personal knowledge of a client.
3. **Task-centric workflows** ‚Äî Work is always visible, trackable, and accountable.
4. **Flexibility through metadata** ‚Äî Use tagging and relationships instead of rigid structures.
5. **Operational transparency** ‚Äî At any moment, leadership can see the true state of the business.

---

### 11. Open Questions (Decide During Design)

- Should all actions originate from tasks?
- How strict should the tagging system be?
- How should document versioning work?
- What defines the "most active policy"?
- How should escalation rules be configured?

---

### 12. V1 Deliverables Summary

The following must be functional in V1:

| Area | Scope |
|------|-------|
| **Command Centre** | Dashboard with personal task queue, overdue tasks, upcoming work, basic stats |
| **Clients** | CRUD for individuals and companies, search, client detail view |
| **Policies** | CRUD linked to clients, status tracking, "active policy" indicator |
| **Claims** | CRUD linked to policies/clients, lifecycle status tracking |
| **Tasks** | Full CRUD, assignment, priority, due dates, status, linked to clients/policies/claims |
| **Documents** | Upload, tag, associate with client/policy/claim/task, download, list view |
| **Notes** | Add notes to clients, inline activity timeline |
| **Activity Timeline** | Chronological feed on client detail (notes, tasks, documents) |
| **Tags** | System tags + user-defined tags for documents |
| **Organisation context** | All data scoped to the active Better Auth organisation |

**Explicitly deferred to V2+:**
- Email / inbox integration
- Folder-based bulk import
- Advanced escalation rules and SLA tracking
- Reporting and analytics
- External API integrations (insurers, underwriters)
